# 第 1 章：什么是 LLM，什么不是

在开始构建 Agent 之前，我们需要先建立对底层引擎——大语言模型（LLM）的正确认知。很多初学者在刚接触 Agent 开发时，容易把 LLM 想象成一个无所不能的“数字大脑”，或者一个已经具备自主行动能力的“机器人”。

这种拟人化的想象在日常使用 ChatGPT 时没有问题，但在工程开发中，它会导致我们对系统的能力边界产生误判。

本章的目标是剥离这些拟人化的想象，从工程视角重新认识 LLM：它到底是什么，它能做什么，以及更重要的，它**不能**做什么。

## 1. LLM 的本质：一个强大的“文本接龙”引擎

在最底层的逻辑上，LLM（Large Language Model）可以被理解为一个极其复杂的概率模型。它的核心任务只有一个：**根据已经给定的上下文（Context），预测下一个最可能出现的词（Token）。**

### 什么是 Token？

在 LLM 的世界里，文本不是按字母或汉字来处理的，而是被切分成一个个的 Token。你可以把 Token 想象成语言的“基本积木”。

- 一个常见的英文单词可能就是一个 Token（比如 "apple"）。
- 一个长单词可能会被切分成多个 Token（比如 "unbelievable" 可能会变成 "un", "believ", "able"）。
- 在中文里，一个汉字通常对应一个或多个 Token，具体取决于模型的词表设计。

### 什么是上下文窗口（Context Window）？

LLM 并不是拥有无限记忆的。它每次进行预测时，只能“看到”有限长度的文本，这个长度限制就是上下文窗口。

你可以把它想象成 LLM 的“短期工作记忆”。如果一段对话超过了上下文窗口的限制，LLM 就会“忘记”最开始的内容。在 Agent 开发中，如何高效地利用和管理这个有限的上下文窗口，是一个核心的工程挑战。

### 什么是采样（Sampling）？

当 LLM 预测下一个 Token 时，它并不是只给出一个绝对确定的答案，而是给出一个概率分布（比如：下一个词是 "苹果" 的概率是 80%，是 "香蕉" 的概率是 15%，是 "汽车" 的概率是 5%）。

采样就是根据这个概率分布来选择最终输出的 Token 的过程。通过调整采样参数（比如 Temperature），我们可以控制模型输出的随机性和创造性。Temperature 越低，模型越倾向于选择概率最高的词，输出越稳定；Temperature 越高，模型越可能选择概率较低的词，输出越具创造性（但也更容易出错）。

## 2. LLM 不是什么？

理解了 LLM 的本质是一个“文本接龙”引擎，我们就能更容易地看清它的能力边界。

### LLM 不是一个自主系统（Autonomous System）

LLM 本身是一个被动的函数：你给它一段输入，它给你一段输出。然后它就停下来了。

它**不会**：

- 自己决定什么时候开始工作。
- 自己决定什么时候停止工作。
- 在没有外部驱动的情况下，主动去执行一个长期的任务。

**对比示例：**

- **LLM 的行为：** 你问它“如何写一个贪吃蛇游戏？”，它会输出一段包含代码和解释的文本，然后结束。
- **Agent 的行为：** 你告诉它“帮我写一个贪吃蛇游戏并运行起来”，它可能会先生成代码，然后尝试保存到文件，接着尝试运行，如果报错了，它会读取错误信息，修改代码，再次运行，直到游戏成功跑起来。

这种“循环执行”、“错误恢复”和“自主决策”的能力，是 Agent Runtime 赋予的，而不是 LLM 原生自带的。

### LLM 不是一个执行环境（Execution Environment）

LLM 只能生成文本，它**不能**直接对现实世界或计算机系统产生影响。

它**不会**：

- 读写你电脑上的文件。
- 执行 Shell 命令。
- 访问互联网获取最新信息。
- 操作数据库。

当你在 ChatGPT 中看到它似乎能“搜索网页”或“运行 Python 代码”时，那是因为 OpenAI 在 LLM 外面包了一层系统。LLM 只是生成了“我要搜索 X”或“我要运行这段代码”的**文本指令**，然后由外部系统去实际执行这些指令，并将结果再作为文本喂回给 LLM。

### LLM 不是一个可靠的知识库（Reliable Knowledge Base）

虽然 LLM 在训练过程中“阅读”了海量的数据，但它并没有像数据库那样精确地存储这些信息。它的知识是压缩在神经网络的权重中的。

这导致了两个问题：

1. **幻觉（Hallucination）：** 当 LLM 遇到它不知道的问题时，它倾向于根据概率“编造”一个看起来很合理的答案，而不是直接说“我不知道”。
2. **知识滞后：** LLM 的知识停留在它训练完成的那一刻。它不知道昨天发生的新闻，也不知道你刚刚修改的代码。

在 Agent 开发中，我们通常不依赖 LLM 自身的知识来解决特定领域的问题，而是通过工具（比如搜索工具、文件读取工具）将准确的外部信息作为上下文提供给它。

## 3. 在 pi-mono 仓库中的映射

在 `pi-mono` 这个项目中，我们非常清晰地划分了 LLM 的职责边界。

如果你查看 `packages/ai` 这个包，你会发现它主要负责解决“如何稳定地和模型说话”这个问题。

它定义了统一的接口（`Api`），处理了不同模型提供商（Provider）的差异，规范了消息格式（`Message`），并处理了流式输出（Streaming）和错误重试。

但它**不包含**任何关于“如何执行工具”、“如何管理长期记忆”或“如何进行多轮循环”的逻辑。这些更高级的职责，被交给了上层的 `packages/agent` 和 `packages/coding-agent`。

在 `packages/ai` 的视角里，LLM 就是一个纯粹的、无状态的文本生成引擎。

## 4. 总结

在进入下一章学习 Agent 之前，请牢记这个心智模型：

**LLM 是一个极其聪明的“文本生成器”，但它是一个“没有手脚”、“没有长期记忆”、“不会自己动”的生成器。**

Agent 开发的核心，就是围绕这个聪明的生成器，为它装上“手脚”（工具）、“记忆”（状态管理）和“马达”（Agent Loop），让它从一个被动的问答机器，变成一个能主动解决问题的系统。
