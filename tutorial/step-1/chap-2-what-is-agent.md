# 第 2 章：什么是 Agent

这一章接在第 1 章《什么是 LLM，什么不是》后面。

上一章我们先把模型放回了它应有的位置：`LLM` 首先是生成器，不是完整系统。现在要继续往上走一层，回答另一个几乎所有初学者都会混淆的问题：

- 聊天助手到底是不是 agent
- workflow 和 agent 到底差在哪
- 既然 `packages/ai` 已经能调模型了，为什么还需要 `packages/agent`

这一章聚焦的是系统里的 `agent runtime` 这一层，也就是 `packages/agent`。后面的第 3 章会继续讲最小 agent 闭环，第 4 章会专门讲工具调用，第 13 章之后会更系统地拆开 agent loop、context transform、steering、follow-up 这些运行时机制。

## 这一章读完后，你应该能说清楚

- 为什么“能聊天”不等于“是 agent”
- 为什么 workflow 很有用，但它和 agent 不是一回事
- agent 的核心不是“更聪明”，而是“能在运行时驱动动作”
- `packages/agent` 为什么存在，而不是让上层直接基于 `packages/ai` 写产品
- 这个仓库里 agent runtime 的最小职责边界是什么

## 1. 先把三个容易混淆的东西拆开

这一章最重要的不是背定义，而是把三个看起来很像、但工程含义完全不同的东西分开：

1. 聊天助手
2. workflow
3. agent

很多“agent 很神奇”的误解，都是因为把这三者混成了一个词。

## 2. 什么是聊天助手

先说最简单的。

聊天助手可以理解成：

“你给一段输入，它给一段输出。”

它的核心任务通常是：

- 回答问题
- 改写文本
- 总结内容
- 生成一段结构化结果

如果把系统看成一条链路，聊天助手往往更接近：

`用户输入 -> 模型 -> 回复`

这里最重要的能力是“生成内容”，而不是“推动任务向前执行”。

这类系统当然也可能很好用，但它通常没有下面这些能力，或者这些能力不属于系统主干：

- 根据中间结果决定下一步做什么
- 调用多个工具与外部世界交互
- 维护一个持续推进的任务状态
- 在失败、中断、补充信息后继续工作

这就是为什么“一个回答得不错的助手”还不能自动等价成 agent。

## 3. 什么是 workflow

`workflow` 可以先理解成：

“流程主要由程序员提前写好，系统按预设步骤执行。”

例如一个固定流程：

1. 读取工单内容
2. 提取关键信息
3. 生成回复草稿
4. 调用发信接口

这个系统里当然可以用到 LLM，但真正决定流程的是程序，而不是模型。

换句话说，workflow 的核心不是“模型自己决定下一步”，而是：

“开发者已经把下一步写死了。”

workflow 为什么重要？因为它有几个非常现实的优点：

- 可预测
- 好测试
- 好审计
- 好控制成本
- 容易解释为什么会这样执行

所以 workflow 不是低级方案。很多工程问题，workflow 反而是更好的方案。

但 workflow 很容易和 agent 混淆，因为它们都可能：

- 调模型
- 调工具
- 处理多步任务

真正的区别在于：

workflow 的决策结构主要写在代码里，agent 的一部分决策结构发生在运行时。

## 4. 什么是 Agent

这一章先给一个够工程化、也够稳定的定义：

`agent` 是一个会在运行时基于当前上下文决定“下一步是继续思考、调用工具、还是结束”的系统。

这里有几个词很关键。

### 4.1 “运行时决定”

这句话是在和 workflow 做对比。

如果系统一定会先执行 A，再执行 B，再执行 C，那更像 workflow。

如果系统会根据当前消息、工具结果、错误状态、额外插话，动态决定：

- 要不要查文件
- 查哪个文件
- 是不是先调用另一个工具
- 收到工具结果后要不要继续
- 任务是否已经结束

那它才开始具有 agent 的味道。

### 4.2 “基于当前上下文”

agent 不是凭空决策。

它依赖的是当前上下文，包括：

- system prompt
- 历史消息
- 工具定义
- 工具执行结果
- 中断或 follow-up 消息
- 运行时注入的额外信息

所以 agent 的行为并不只是“模型聪不聪明”，还强烈依赖上下文是如何组织的。

### 4.3 “继续思考、调用工具、还是结束”

这句话是在说明 agent 和普通聊天助手的边界。

普通聊天助手最核心的产物是“一段回复”。

agent 最核心的产物往往不是一次回复，而是一个闭环过程：

`消息 -> 模型 -> 工具 -> 工具结果 -> 再次调用模型 -> 结束或继续`

这也是为什么后面我们会单独讲 agent loop。没有 loop，就很难说你真的有了一个 agent runtime。

## 5. Agent 不是“更聪明的聊天”

初学者最容易产生的误解是：

“agent 就是一个更强、更聪明、更会推理的聊天模型。”

这不准确。

从工程角度看，agent 的关键增量通常不是“多了一点智能”，而是“多了一套可运行的控制结构”。

这套控制结构至少解决几件事情：

- 如何把用户消息加入上下文
- 什么时候发起模型调用
- 如何识别模型发出的工具调用
- 如何执行工具
- 如何把工具结果再写回上下文
- 什么时候开启下一轮
- 什么时候停止
- 失败或中断时系统该处在什么状态

如果这些都没有，只是把 prompt 写得更长、回复更花哨，那通常还只是聊天系统。

## 6. 一个具体对比：聊天助手、workflow、agent 到底差在哪

### 场景：帮用户分析仓库并修改一个配置文件

#### 方案一：聊天助手

用户说：“帮我看看这个项目为什么启动失败。”

系统只做一件事：

- 把用户的话发给模型
- 返回一段建议

模型也许会说“请检查 `package.json` 和环境变量”，但系统本身不会真的去读文件，也不会执行命令。

这就是聊天助手。

#### 方案二：workflow

程序员提前写好流程：

1. 读取 `package.json`
2. 读取 `.env`
3. 运行一条检查命令
4. 把结果拼成上下文交给模型总结

这个系统已经能做事了，但顺序是预先固定的。它并不会根据中间结果自由改变路径。

这就是 workflow。

#### 方案三：agent

系统先让模型看当前任务和可用工具，然后在运行时决定：

- 先搜启动脚本
- 再读某个配置文件
- 发现需要继续查另一个文件
- 决定执行一个命令确认假设
- 拿到结果后再决定是否修改文件
- 修改后再决定是否继续验证

这时系统的关键不是“某一步用了 LLM”，而是：

模型和 runtime 一起决定了多步动作如何推进。

这就是 agent。

## 7. 为什么 Agent 必须有工具

不是所有 agent 都必须有外部工具，但对这个仓库想解决的问题来说，工具几乎是决定性的。

原因很简单：

没有工具，模型只能生成文本。

而真实任务往往要求系统与外部世界交互，例如：

- 读文件
- 写文件
- 搜索代码
- 执行命令
- 请求网页
- 保存状态

一旦模型能请求这些动作，系统的性质就变了。

它不再只是“描述要做什么”，而是开始“推动事情发生”。

这也是为什么在 `packages/agent` 里，工具不是一个附加小功能，而是主干的一部分。`AgentTool`、工具执行事件、`beforeToolCall`、`afterToolCall` 这些能力都围绕这个核心展开。

对应代码可以先看：

- `AgentTool` 定义在 `packages/agent/src/types.ts`
- 工具执行主逻辑在 `packages/agent/src/agent-loop.ts`

## 8. 为什么有了 `packages/ai`，还需要 `packages/agent`

这是这一章最关键的架构问题。

上一章讲过，`packages/ai` 负责把不同 provider 的差异压平，让上层拿到统一的模型、消息、事件流和 tool-calling 语义。

但即使这层已经存在，它解决的仍然只是：

“怎么更稳定地和模型说话。”

它没有解决下面这些问题：

- 多轮循环何时继续
- 工具何时执行
- 工具结果何时回写
- 用户中途插话如何处理
- 任务完成后 follow-up 如何排队
- 当前 agent 运行状态如何暴露给 UI

这些问题都不是“模型接口层”的问题，而是“运行时控制层”的问题。

这正是 `packages/agent` 存在的原因。

如果没有这一层，上层产品就得自己反复重写：

- agent loop
- 工具执行策略
- 状态管理
- 事件分发
- 中断和继续逻辑

这会导致复杂度继续向上传染到 `packages/coding-agent`、`packages/web-ui` 或其他具体产品。

所以这两个包解决的是不同层的问题：

- `packages/ai` 解决“统一 LLM 接口”
- `packages/agent` 解决“把统一的 LLM 接口变成可运行的 agent runtime”

## 9. 这个仓库里，Agent 的最小闭环长什么样

看 `packages/agent/src/agent-loop.ts`，你会发现这个仓库对 agent 的理解非常朴素，也非常工程化。

它的最小闭环大致是：

1. 接收新的 prompt 或从已有上下文继续
2. 发出生命周期事件
3. 把 `AgentMessage[]` 转成 LLM 可理解的 `Message[]`
4. 调用 `streamSimple()` 获取 assistant 响应
5. 如果响应里有工具调用，就执行工具
6. 把工具结果作为 `toolResult` 消息写回上下文
7. 再次进入下一轮
8. 没有工具调用且没有 follow-up 时结束

这条链路很重要，因为它说明了一个事实：

agent 不是单个函数，不是单条 prompt，也不是单次调用。

agent 是一个会维护状态、处理事件、推进回合的系统。

## 10. `AgentMessage` 为什么不是直接复用 LLM Message

这是 `packages/agent` 非常值得初学者注意的一点。

在 `packages/agent/src/types.ts` 里，`AgentMessage` 不是简单等于底层 LLM 的 `Message`，而是：

- 包含标准 LLM 消息
- 允许应用通过 declaration merging 扩展自定义消息类型

为什么要多这一层？

因为 runtime 面对的世界，比模型理解的世界更大。

模型只需要理解：

- `user`
- `assistant`
- `toolResult`

但一个真实系统可能还需要：

- UI 状态消息
- artifact 消息
- 通知消息
- 产品层自己的结构化消息

如果直接把所有上层消息都强行塞进 LLM 消息类型，边界会变得非常混乱。

所以 `packages/agent` 专门保留了一层更贴近应用的消息模型，然后通过 `convertToLlm` 在真正调用模型前做转换。

这就是为什么 README 里会强调：

`AgentMessage vs LLM Message`

这个对比不只是类型设计细节，它代表的是 runtime 和模型边界之间的一道隔离带。

## 11. Agent 为什么要有事件流，而不只是最终结果

如果你把 agent 想成真实系统，而不是一次函数调用，这个问题就很好回答。

UI、日志、状态同步、调试、恢复，都不只关心“最后一句话是什么”。

它们还关心：

- agent 什么时候开始
- 当前是哪一轮
- assistant 是否正在流式输出
- 哪个工具开始执行了
- 工具有没有中间进度
- 工具什么时候结束
- 当前轮次是否结束
- 整个运行是否结束

这就是为什么 `packages/agent/src/types.ts` 里定义了完整的 `AgentEvent` 联合类型，也为什么 README 专门把 event flow 单独拿出来解释。

对比一下：

- 普通聊天接口更像“返回一个结果”
- agent runtime 更像“发出一串描述系统演化过程的事件”

这个区别非常重要，因为后面的 `packages/tui` 和 `packages/web-ui` 都是建立在这类事件之上的。

## 12. `Agent` 类和低层 loop 有什么区别

这一章不需要把实现细节全吃透，但要先建立一个边界感。

`packages/agent` 同时暴露了两层东西：

- `agentLoop()` / `agentLoopContinue()`
- `Agent` 类

它们不是重复实现，而是抽象层级不同。

### 12.1 低层 loop

在 `packages/agent/src/agent-loop.ts` 里，`agentLoop()` 更接近一个可观察的运行引擎。

它负责：

- 推进回合
- 发出事件
- 执行工具
- 写回 tool result

这层适合想直接控制流程的人。

### 12.2 `Agent` 类

在 `packages/agent/src/agent.ts` 里，`Agent` 则是在 loop 之上的状态化封装。

它额外解决的是：

- 当前 transcript 由谁持有
- 当前是否正在运行
- 正在流式输出的消息是什么
- 哪些 tool call 仍在执行
- 外部如何订阅事件
- 如何排队 steering 和 follow-up 消息
- `prompt()`、`continue()`、`abort()`、`waitForIdle()` 这些产品更常用的交互方式

所以可以把它理解成：

- `agentLoop` 更像 runtime engine
- `Agent` 更像可直接拿来接 UI 和应用状态的 runtime facade

## 13. 为什么说 Agent 的核心是“控制结构”

现在可以把前面的内容压成一个更稳的判断标准。

如果一个系统只有：

- prompt
- 模型调用
- 返回文本

那它主要还是聊天系统。

如果一个系统多了：

- 上下文转换
- 工具执行
- 工具结果回写
- 回合循环
- 中断与继续
- 运行状态与事件流

那它才真正长出了 agent runtime。

换句话说，agent 这个词在工程里真正有意义的地方，不是“模型更像人”，而是“系统更像一个能持续推进任务的状态机”。

这也是为什么本章会把重点放在 `packages/agent`，而不是继续讨论模型能力本身。

## 14. 用仓库代码把这个概念落下来

这一章最值得读的几个入口如下。

### `packages/agent/README.md`

这是理解整个包职责边界最快的地方。它明确告诉你，这一层是：

- stateful agent
- tool execution
- event streaming

而且 built on `@mariozechner/pi-ai`。

这句话本身已经说明了边界：

- 下层不是 UI
- 下层也不是 provider SDK
- 它正好处在模型接口层和产品层之间

### `packages/agent/src/types.ts`

这个文件最值得关注的不是细枝末节，而是它定义了这层系统到底认为什么叫“agent runtime 的公共协议”：

- `AgentMessage`
- `AgentState`
- `AgentTool`
- `AgentContext`
- `AgentEvent`
- `AgentLoopConfig`

这些类型共同描述了运行时的世界观。

### `packages/agent/src/agent-loop.ts`

这个文件是最直接的“agent 到底如何跑起来”的答案。

其中最关键的不是某个技巧，而是那条控制链：

- 收消息
- 调模型
- 看是否有 tool call
- 执行工具
- 写回结果
- 再来一轮

这是本仓库里 agent 的骨架。

### `packages/agent/src/agent.ts`

这个文件说明了为什么“有 loop”还不够。

真实产品还需要：

- 可持有状态的对象
- 可订阅事件的接口
- 中断和继续
- 外部注入 steering / follow-up

也就是说，从 loop 到产品之间，还有一层“把运行时做成可用对象”的工作。

## 15. 本章最容易出现的误解

### 误解一：只要模型会调工具，它就是 agent

不完全对。

模型会产生 `toolCall` 只是一个开始。真正让系统变成 agent 的，是外部 runtime 知道如何：

- 验证参数
- 执行工具
- 写回结果
- 控制下一轮

少了这些，工具调用只是输出格式，不是可运行系统。

### 误解二：workflow 比 agent 更“笨”

不对。

workflow 只是把决策更多放在代码里，而不是运行时里。很多高可靠场景，workflow 反而更合适。

真正该问的不是“哪个更高级”，而是：

“这个问题需要多少运行时决策自由度？”

### 误解三：agent 只是 prompt engineering 的延伸

不对。

prompt 很重要，但 agent 的难点往往在边界管理：

- 状态
- 工具
- 事件
- 中断
- 错误恢复
- 成本与可控性

这些都不是单靠 prompt 能解决的。

### 误解四：`packages/agent` 只是对 `packages/ai` 的薄封装

不对。

如果只是薄封装，它就不需要自己的消息类型、状态、事件、工具执行策略、steering、follow-up 和 `Agent` 类。

它存在的意义，正是把“统一模型接口”提升成“可运行的 agent runtime”。

## 16. 本章小结

这一章你至少要带走下面六个结论：

1. 聊天助手的核心是生成回复，agent 的核心是推进任务。
2. workflow 的决策主要写在代码里，agent 的一部分决策发生在运行时。
3. agent 不等于“更聪明的模型”，而是“带有控制结构的系统”。
4. 工具让 agent 从“会说”变成“会做”，但工具调用本身还不等于完整 agent。
5. `packages/ai` 解决模型接口统一，`packages/agent` 解决运行时闭环。
6. 在这个仓库里，agent 最重要的骨架是消息、循环、工具执行、状态和事件流。
