# AgentSession 事件队列理解笔记

这份笔记只解释几个最容易绕晕的点：

- `_handleAgentEvent()` 是怎么调用 `_processAgentEvent()` 的
- `_agentEventQueue` 到底是什么
- `Promise.resolve()` 在这里是什么意思
- 为什么要把 `_processAgentEvent()` 同时挂到 `.then()` 的成功和失败回调里
- 那段关于 retry 的长注释到底在防什么问题

## 1. 整体调用链

先看最外层关系：

1. `AgentSession` 构造时，调用 `this.agent.subscribe(this._handleAgentEvent)`
2. 底层 `Agent` 每发出一个 `AgentEvent`，就会同步调用这个 listener
3. 这个 listener 就是 `_handleAgentEvent(event)`
4. `_handleAgentEvent(event)` 不直接处理事件，而是把 `_processAgentEvent(event)` 排进一个 Promise 队列

也就是说：

- `_handleAgentEvent()` 负责“接住事件”
- `_processAgentEvent()` 负责“真正处理事件”
- `_agentEventQueue` 负责“保证这些事件按顺序一个一个处理”

## 2. `_agentEventQueue` 不是事件数组

最容易误解的一点是：`_agentEventQueue` 名字里有 `Queue`，但它不是一个数组，不保存事件列表。

它的定义是：

```ts
private _agentEventQueue: Promise<void> = Promise.resolve();
```

所以它本质上只是一个 `Promise<void>`。

更准确地说，它表示的是：

“前面已经排队的事件，什么时候全部处理完成。”

可以把它理解成“当前队尾的完成凭证”。

## 3. `Promise.resolve()` 是什么意思

`Promise.resolve()` 会返回一个“已经成功完成”的 Promise。

在这里它的作用很简单：

- 作为空队列的起点
- 表示一开始前面没有任务
- 所以第一个事件可以直接接到它后面

也可以理解成：

```ts
_agentEventQueue = 已完成
```

意思就是：当前没有任何排队中的事件。

## 4. `_handleAgentEvent()` 是怎么“调用” `_processAgentEvent()` 的

关键代码是：

```ts
this._agentEventQueue = this._agentEventQueue.then(
  () => this._processAgentEvent(event),
  () => this._processAgentEvent(event),
);
```

这不是“立刻执行 `_processAgentEvent(event)`”，而是：

1. 等当前队尾 Promise 结束
2. 再执行当前事件的 `_processAgentEvent(event)`
3. 然后把新的 Promise 再存回 `_agentEventQueue`

所以这是一种“把任务挂到队尾”的写法。

## 5. 为什么这样能形成串行队列

假设连续收到 3 个事件：

1. `message_start`
2. `message_end`
3. `agent_end`

初始状态：

```ts
q0 = Promise.resolve()
```

第一个事件到来：

```ts
q1 = q0.then(() => process(message_start), () => process(message_start))
```

第二个事件到来：

```ts
q2 = q1.then(() => process(message_end), () => process(message_end))
```

第三个事件到来：

```ts
q3 = q2.then(() => process(agent_end), () => process(agent_end))
```

这时执行顺序就被锁定成：

```ts
process(message_start)
process(message_end)
process(agent_end)
```

即使这些事件几乎同时到达，`_processAgentEvent()` 也不会并发执行，而是一个接一个执行。

所以 `_agentEventQueue` 的作用就是：

- 不让事件处理并发乱序
- 让所有事件按收到的顺序串行处理

## 6. 为什么成功和失败回调都要写

如果只写成功回调：

```ts
this._agentEventQueue = this._agentEventQueue.then(
  () => this._processAgentEvent(event),
);
```

那一旦前一个事件处理失败，`_agentEventQueue` 就会变成 rejected。

后面再来的事件继续接这个 Promise 时，因为前面的 Promise 已经失败了，默认不会再走成功分支。这样后续事件可能全部不处理了。

而现在的写法是：

```ts
this._agentEventQueue = this._agentEventQueue.then(
  () => this._processAgentEvent(event),
  () => this._processAgentEvent(event),
);
```

含义是：

- 前一个事件成功了，处理当前事件
- 前一个事件失败了，也处理当前事件

所以它在表达：

“不管前面那个事件成没成功，当前这个事件都要继续进入队列执行。”

这就是为什么它要把同一个 `_processAgentEvent(event)` 挂到成功和失败两边。

## 7. `catch(() => {})` 是干什么的

后面还有一句：

```ts
this._agentEventQueue.catch(() => {});
```

它的主要作用是避免未处理的 Promise rejection 冒出去。

可以把它理解成：

- 这个队列内部即使某次处理失败了
- 失败会留在队列语义里
- 但不会变成“没人接住的异常”往外炸

结合前面的 `.then(success, failure)`，整体目标就是：

- 事件一个个处理
- 某个事件处理失败不影响后面的事件继续排队

## 8. 为什么不直接在 `_handleAgentEvent()` 里 `await _processAgentEvent()`

如果 `_handleAgentEvent()` 里直接处理：

```ts
await this._processAgentEvent(event)
```

那每次 listener 被调用时都会直接进入当前处理逻辑。

问题在于：

- 事件可能来得很快
- 某些处理可能比较慢
- 如果没有这个 Promise 队列，事件处理可能交错在一起
- 一些依赖顺序的状态更新、扩展事件、持久化逻辑就容易乱

所以这里不是单纯想“异步执行”，而是想“异步但严格串行执行”。

## 9. 那段 retry 注释到底在说什么

原注释的核心意思是：

“创建 retry promise 这件事，必须同步做，不能等排队后的 `_processAgentEvent(agent_end)` 再做，否则等待方可能会错过这次 retry。”

这里有一个竞态：

1. `agent_end` 事件已经到了
2. 但因为前面还有别的事件在排队，`_processAgentEvent(agent_end)` 还没开始执行
3. 与此同时，外层代码在 `agent.prompt()` 返回后，立刻调用 `waitForRetry()`
4. 如果 `_retryPromise` 这时还没创建，`waitForRetry()` 会以为“没有 retry 正在进行”，于是直接返回

这就会导致：

- 实际上稍后会启动一次 retry
- 但等待方已经提前放行了
- 等于错过了这次正在路上的 retry

所以 `_handleAgentEvent()` 一接到事件，就先同步执行：

```ts
this._createRetryPromiseForAgentEnd(event);
```

只有这样，`waitForRetry()` 才能及时看到 `_retryPromise` 已经存在。

## 10. 一句话总结

如果只记一句，可以记这个：

`_agentEventQueue` 是“前面事件全部处理完成”的 Promise，`_handleAgentEvent()` 每收到一个新事件，就把 `_processAgentEvent(event)` 继续接到这个 Promise 后面，从而形成一个不会因为单次失败而中断的串行事件队列。

## 11. 再压缩成最短版本

- `Promise.resolve()`：空队列起点
- `_agentEventQueue`：当前队尾 Promise
- `.then(..., ...)`：把当前事件接到队尾
- 成功失败都接 `_processAgentEvent`：前一个事件炸了，后一个也照常处理
- 先同步创建 `_retryPromise`：避免 `waitForRetry()` 错过即将开始的自动重试
