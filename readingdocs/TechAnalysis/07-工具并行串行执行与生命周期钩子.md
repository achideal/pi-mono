# 07 · Tool 并行 / 串行执行 + before / afterToolCall 钩子

> 位置：`packages/agent/src/agent-loop.ts` 的 `executeToolCalls` / `executeToolCallsParallel` / `executeToolCallsSequential` / `prepareToolCall` / `finalizeExecutedToolCall`
>
> 提炼点：**把"多 tool_call 怎么跑"抽象成 2 条可切换路径，同时在每个 tool call 生命周期上留出两个精准钩子——`beforeToolCall`（权限门）+ `afterToolCall`（审计/重写）。并行模式保留"源顺序"的 ToolResult 发射次序，从而不破坏 LLM 协议。**

---

## 1. 需求：现代 LLM 会一次吐多个 tool_call

Claude / GPT-5 / Gemini 都支持一次 assistant 消息里吐 N 个 tool_call。最朴素的实现是串行：

```
for tc in toolCalls: run(tc); record(result);
```

正确但慢。如果每个 tool 都是网络请求、每个要 2 秒，10 个调用要 20 秒，而 LLM 那边只等 1 个结果就能继续推理。

另一方面，某些工具**不能并行**——比如 `write_file` + `edit_file` 碰到同一个文件会竞态；`bash` 的 cd 会影响后续调用。一刀切"全并行"反而 break 用户。

pi-agent 的设计：**用户可以选 `sequential` 或 `parallel`，并行路径下仍然严格保留"发射 toolResult 消息的顺序"，让 LLM 以为一切都按它吐出来的次序执行。**

---

## 2. 两条路径的顶层分派

```ts
async function executeToolCalls(...) {
  const toolCalls = assistantMessage.content.filter((c) => c.type === "toolCall");
  if (config.toolExecution === "sequential") {
    return executeToolCallsSequential(...);
  }
  return executeToolCallsParallel(...);
}
```

默认值是 `parallel`（见 `Agent` 构造函数）。切换成本是一个字段，`agent.toolExecution = "sequential"` 运行时也能切。

## 3. 统一的"生命周期"抽象

无论哪条路径，每个 tool_call 都要经过 4 步：

```
emit tool_execution_start
  ↓
prepareToolCall()       ← 校验 + beforeToolCall 钩子
  ↓
executePreparedToolCall() ← 真正调用 tool.execute
  ↓
finalizeExecutedToolCall() ← afterToolCall 钩子 → emit tool_execution_end + toolResult 消息
```

这 4 步被抽出成独立函数，让 Sequential 和 Parallel 路径**只差一个 Promise.all**。

### 3.1 `prepareToolCall`

```ts
async function prepareToolCall(...): Promise<PreparedToolCall | ImmediateToolCallOutcome> {
  const tool = currentContext.tools?.find((t) => t.name === toolCall.name);
  if (!tool) return { kind: "immediate", result: createErrorToolResult(`Tool ${toolCall.name} not found`), isError: true };

  try {
    const preparedToolCall = prepareToolCallArguments(tool, toolCall);
    const validatedArgs = validateToolArguments(tool, preparedToolCall);

    if (config.beforeToolCall) {
      const beforeResult = await config.beforeToolCall(
        { assistantMessage, toolCall, args: validatedArgs, context: currentContext },
        signal,
      );
      if (beforeResult?.block) {
        return {
          kind: "immediate",
          result: createErrorToolResult(beforeResult.reason || "Tool execution was blocked"),
          isError: true,
        };
      }
    }
    return { kind: "prepared", toolCall, tool, args: validatedArgs };
  } catch (error) {
    return { kind: "immediate", result: createErrorToolResult(..), isError: true };
  }
}
```

几件事一口气做完：

1. **查 tool 是否存在**：找不到直接变成 "immediate" 错误，绕过后续执行。
2. **prepareArguments**：tool 自己可以提供一个 shim，把"模型吐的 snake_case / 老字段名"改成新的 schema 形状。向后兼容工具升级。
3. **validateToolArguments**（AJV + TypeBox，详见第 8 篇）：只在这里发生一次。下游执行时 args 是已经被类型收窄过的。
4. **beforeToolCall 钩子**：接受 block 决策或通过。block 同样变成 "immediate" 错误。
5. **任何异常**都被 try/catch 成 immediate 错误。**不会抛穿到主循环**。

### 3.2 `executePreparedToolCall`

```ts
async function executePreparedToolCall(prepared, signal, emit): Promise<ExecutedToolCallOutcome> {
  const updateEvents: Promise<void>[] = [];
  try {
    const result = await prepared.tool.execute(
      prepared.toolCall.id,
      prepared.args as never,
      signal,
      (partialResult) => {
        updateEvents.push(Promise.resolve(emit({ type: "tool_execution_update", ..., partialResult })));
      },
    );
    await Promise.all(updateEvents);
    return { result, isError: false };
  } catch (error) {
    await Promise.all(updateEvents);
    return { result: createErrorToolResult(error.message), isError: true };
  }
}
```

值得记下来的实现细节：

- **partialResult 回调的 emit 是被收集的**，等 execute 结束统一 `await Promise.all`。这保证了"partial 事件不会在 tool_execution_end 之后才到达 listener"。
- tool 内部的异常被 catch 转换成 error result，**也没有漏掉已经排队的 partial events**。
- execute 的第三个参数是 AbortSignal，tool 作者要自己 honor（例如把 signal 传给 fetch），这是每个工具作者应当遵守的契约。

### 3.3 `finalizeExecutedToolCall`（`afterToolCall` 钩子在这）

```ts
if (config.afterToolCall) {
  const afterResult = await config.afterToolCall({ ..., result, isError, context }, signal);
  if (afterResult) {
    result = {
      content: afterResult.content ?? result.content,
      details: afterResult.details ?? result.details,
    };
    isError = afterResult.isError ?? isError;
  }
}
return await emitToolCallOutcome(prepared.toolCall, result, isError, emit);
```

注释里写得很清楚：**"No deep merge"**。returned `content` 会整个替换 content 数组，returned `details` 整个替换 details——避免"部分合并"带来的不可预期。

## 4. `beforeToolCall`：权限门 / 沙盒

典型用法：

- **要用户二次确认**：调用 `bash` 之前弹窗，等用户点"同意"再返回 `undefined`，否则返回 `{ block: true, reason: "User denied" }`。
- **路径保护**：`write_file` 的 path 不能指向 `/etc/`，block 掉。
- **预算/计费**：每个 tool 调用都计入预算，超了就 block。
- **速率限制**：单位时间内某个 tool 最多 N 次。

重要点：block 时返回的 `reason` 会直接成为 tool_result 的 text。LLM **会看到"你为什么被拒"**，可以下一步自我修正。这比静默失败好得多。

---

## 5. `afterToolCall`：审计 / 改写 / 注入 metadata

典型用法：

- **加摘要标签**：审计日志需要标记 `{ audited: true, reviewer: "gpt-4o-mini" }`，塞到 details 里不影响 content。
- **截断超长输出**：grep 跑出 100k 行，在 afterToolCall 里把 content 替换成"前 2000 行 + 省略说明"，防止填满 context。
- **redact 敏感信息**：tool 返回里有密码/token，用正则替换成 `***`。
- **二次判错**：某工具返回 `isError: false` 但内容里有 "ERROR:"，可以在这里翻转 `isError`。

它是"离 LLM 最近的"一道加工工序，是 Extension 集成最常用的钩子。

## 6. Parallel 路径的顺序保持

这是 Parallel 的点睛之笔：

```ts
async function executeToolCallsParallel(...) {
  const results: ToolResultMessage[] = [];
  const runnableCalls: PreparedToolCall[] = [];

  // (1) Preflight 串行：发 tool_execution_start + 跑 prepareToolCall(含 beforeToolCall)
  for (const toolCall of toolCalls) {
    await emit({ type: "tool_execution_start", ... });
    const preparation = await prepareToolCall(...);
    if (preparation.kind === "immediate") {
      results.push(await emitToolCallOutcome(toolCall, preparation.result, preparation.isError, emit));
    } else {
      runnableCalls.push(preparation);
    }
  }

  // (2) 真正的 execute 并行启动
  const runningCalls = runnableCalls.map((prepared) => ({
    prepared,
    execution: executePreparedToolCall(prepared, signal, emit),
  }));

  // (3) Finalize 按源顺序串行 await
  for (const running of runningCalls) {
    const executed = await running.execution;
    results.push(await finalizeExecutedToolCall(currentContext, assistantMessage, running.prepared, executed, config, signal, emit));
  }

  return results;
}
```

三段要分清：

1. **Preflight 串行**：beforeToolCall 串行跑。这让权限门可以基于前一个 tool 的决策来判断（例如"已经批准了 bash，后面都免审"）。
2. **execute 并行**：真正耗时的 I/O 全部并行启动（通过 `map` 立刻把所有 execute 吃进事件循环，但**不 await**）。
3. **按源顺序串行 finalize**：
   - 即使后面的 tool 先跑完，也要先等前面的 finalize。
   - `emit({ type: "tool_execution_end" })` 和"往 messages 里 push toolResult"都按原顺序发。
   - LLM 拿到的 messages 顺序永远等于它自己吐出来的 tool_call 顺序。不会因为某个 tool 先跑完就被插队。

这意味着：**Provider 协议的顺序约束不会被破坏，但实际执行拿到了并发加速**。这种"异步启动 + 顺序收尾"是异步代码里最漂亮的模式之一。

## 7. Sequential 路径：最简单的对称版本

```ts
async function executeToolCallsSequential(...) {
  const results: ToolResultMessage[] = [];
  for (const toolCall of toolCalls) {
    await emit({ type: "tool_execution_start", ... });
    const preparation = await prepareToolCall(...);
    if (preparation.kind === "immediate") {
      results.push(await emitToolCallOutcome(toolCall, preparation.result, preparation.isError, emit));
    } else {
      const executed = await executePreparedToolCall(preparation, signal, emit);
      results.push(await finalizeExecutedToolCall(..., executed, ...));
    }
  }
  return results;
}
```

本质是 parallel 的"每一步都 await"版本。业务代码完全没有 if/else 污染，因为两条路径用的是同一套"步骤函数"。

这一点太容易被低估：**只要你把 "一件事的每个生命周期步骤" 抽成无状态函数，不同并发策略就只是 `for` / `map + Promise.all` 的差别**。Parallel 和 Sequential 这两个 90 行函数加起来才 180 行，既不重复又都读得懂。

## 8. 可以直接借鉴的套路

1. **把 tool 生命周期拆成 4 个纯步骤**：start / prepare / execute / finalize。之后并发策略只是编排这些步骤。
2. **所有错误变 immediate outcome**：主循环永远不 catch 异常，每种失败都是一个合法的 toolResult。
3. **before / after 钩子放在最精确的时机**：before 在参数校验之后、执行之前；after 在执行之后、emit 之前。改写的权力最大，边界最清。
4. **block 的 reason 直接作为 tool_result 返回给 LLM**：不要静默失败，让模型能自我修正。
5. **并行路径"启动并发 + 按源顺序收尾"**：Promise 早启动立刻被事件循环消费，但 await 顺序决定事件顺序。
6. **partial event 的 emit 被收集后统一 await**：避免"partial 晚于 end"的乱序。

任何"批处理 N 个异步任务 + 对外需要严格顺序"的系统（消息队列、事件流处理、测试 runner）都适用这个模板。把 hooks 挪到你的业务语义上（"before each item"、"after each item"），剩下的编排就是本文的那几段。
