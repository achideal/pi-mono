# 09 · TUI 差分渲染 + CSI 2026 同步输出：让终端 UI 零闪烁

> 位置：`packages/tui/src/tui.ts` 的 `doRender()`、`scheduleRender()`、`compositeOverlays()`
>
> 提炼点：**在一个文本终端里实现高频刷新、可折叠叠加、跟随光标滚动的 UI 的关键是三件事——(1) 把 render 产物当作"虚拟行缓冲"做 diff；(2) 用 CSI 2026 把一批 ANSI 指令包成原子更新；(3) 精确区分 4 种场景（首次 / 宽度变化 / 追加行 / 行内刷新）分别选择最便宜的写出路径。**

---

## 1. 问题：终端刷新有天然的"撕裂"

终端的绘制模型是"把字符 + ANSI 控制序列往 stdout 里泼"，如果你每次刷新都：

1. 清屏 (`\x1b[2J`)
2. 移动到 `(0,0)` (`\x1b[H`)
3. 重新 print 所有内容

那**每次刷新间**终端都会短暂是空白，然后内容逐行被画出来。用户视觉上就是"整页闪一下"。

另外，终端的滚动缓冲区（scrollback）是很宝贵的——一旦你清屏了，之前的日志就消失。如果 TUI 盖在 shell 已有的 scrollback 上，你更不能粗暴地清屏，否则用户按上键翻历史会看到一堆空行。

pi-tui 的 TUI 类用 ~1200 行代码解决了这些问题，核心思路非常值得学习。

---

## 2. 关键 API：`requestRender` + 批量合并

外部所有触发渲染的路径最终都调用：

```ts
requestRender(force = false): void {
  if (this.renderRequested) return;
  this.renderRequested = true;
  process.nextTick(() => this.scheduleRender());
}

private scheduleRender(): void {
  if (this.stopped || this.renderTimer || !this.renderRequested) return;
  const elapsed = performance.now() - this.lastRenderAt;
  const delay = Math.max(0, TUI.MIN_RENDER_INTERVAL_MS - elapsed);   // 16ms 节流
  this.renderTimer = setTimeout(() => {
    this.renderTimer = undefined;
    if (this.stopped || !this.renderRequested) return;
    this.renderRequested = false;
    this.lastRenderAt = performance.now();
    this.doRender();
    if (this.renderRequested) this.scheduleRender();   // 期间又有请求就继续
  }, delay);
}
```

三点设计很精致：

1. **`renderRequested` 布尔值去重**：100 次 `requestRender` 聚合成 1 次实际 render。
2. **`process.nextTick` 让同一个事件循环里的多次 state change 都有机会合并**。
3. **16ms (`MIN_RENDER_INTERVAL_MS`) 节流**：相当于 60fps 的上限，避免 stream 刷到百帧导致 CPU 打满或终端掉字。
4. **"空闲期又有请求就继续"**：render 结束后检查 `renderRequested`，保证新状态不被吞。

## 3. 三种渲染策略

在 `doRender()` 里，根据前后帧对比选择：

| 场景 | 写出方式 | 代价 |
| --- | --- | --- |
| **首次渲染** (`previousLines.length === 0`) | 直接按行 print，**不清屏** | 最便宜，不破坏 scrollback |
| **宽度变化 / viewport 上移 / scrollback 异常** | `\x1b[2J\x1b[H\x1b[3J` 全清屏 + 全量重画 | 最重，但必然要重算 wrap |
| **普通更新** | 差分：找出 `firstChanged`/`lastChanged`，只重画这段 | 最省，常见场景命中率高 |

这种策略选择是 TUI 能做到"流式 agent 日志几十 KB/s 吐字"仍不闪烁的关键。

### 3.1 第一次渲染不清屏

```ts
if (this.previousLines.length === 0 && !widthChanged && !heightChanged) {
  logRedraw("first render");
  fullRender(false);  // 注意参数 false = 不 clear
  return;
}
```

`fullRender(false)` 里直接 `\r\n` 打印所有行，**不碰 scrollback**。这样 `pi` 被启动前用户 shell 的历史都还在，他往上翻还能看到开机前的内容。这一条多少次被忽视。

### 3.2 宽度变化必须整屏重来

```ts
if (widthChanged) { fullRender(true); return; }
```

因为文本 wrap 会完全变，你甚至没法可靠地知道"屏上的哪一行对应你内部的哪一行"。暴力清屏是最可靠的选项。

### 3.3 差分路径

```ts
let firstChanged = -1, lastChanged = -1;
const maxLines = Math.max(newLines.length, this.previousLines.length);
for (let i = 0; i < maxLines; i++) {
  const oldLine = i < this.previousLines.length ? this.previousLines[i] : "";
  const newLine = i < newLines.length ? newLines[i] : "";
  if (oldLine !== newLine) {
    if (firstChanged === -1) firstChanged = i;
    lastChanged = i;
  }
}
```

精确到"第 i 行字符串是否相等"。因为每个 Component 返回的是数组 `string[]`，平面的行比较非常轻量。

然后根据 `firstChanged / lastChanged`：

- 把硬件光标移到 `firstChanged` 行。
- `\x1b[2K` 清当前行 + 写新内容。
- 只到 `lastChanged` 就停，**不刷后面的行**（即使语义上"没变"，也省一笔写出）。
- 如果新内容比旧短，拉下去把多出来的行 `\x1b[2K` 清掉。

这种"只动改过的行"的策略非常重要：一个 loading 动画只有 spinner 变了，每帧只重画那 1 行，终端几乎无感。

## 4. CSI 2026 同步输出：让一整帧原子化

终端的"撕裂"不仅来自"先清再画"，也来自"一行一行 flush"。即使一帧只重画 10 行，如果 OS / 终端把这 10 行分两个 tick 显示出来，用户仍然看到"半帧"。

CSI 2026 是 VT 协议相对新的扩展：

- `\x1b[?2026h`：Begin Synchronized Output —— 接下来的输出先在终端内部 buffer 住，暂不渲染到屏幕。
- `\x1b[?2026l`：End Synchronized Output —— 把 buffer 好的内容一次性渲染到屏幕。

pi-tui 的 doRender 把 buffer 构造完整后一次 write：

```ts
let buffer = "\x1b[?2026h";
// ... 拼接所有光标移动 + 清行 + 写新内容 ...
buffer += "\x1b[?2026l";
this.terminal.write(buffer);
```

在支持 2026 的终端里（Kitty / WezTerm / Ghostty / Alacritty / iTerm2 / Windows Terminal 等），上面那整堆动作**对用户来说是一帧 atomic 的**。在不支持的终端里会被静默忽略（序列本身是"未知的屏幕控制"），也不会损坏输出。

这是一个"有它加分，没它无害"的 feature flag 级优化。值得所有 TUI 作者知道。

## 5. `compositeOverlays`：把叠加层画在虚拟行上

Overlay（模态框、下拉菜单）是图形 UI 的家常便饭，但在终端里做 overlay 很棘手：终端是 1D 的字符流，没有"图层"。pi-tui 的做法是先在 JS 里把所有东西合成到**虚拟的 string[]**，再统一进入差分路径：

```ts
private compositeOverlays(lines: string[], termWidth, termHeight): string[] {
  if (this.overlayStack.length === 0) return lines;
  const result = [...lines];

  const rendered = ...;  // 每个 overlay pre-render 出自己的 lines + 位置

  // 用 terminal 行数 padding 到足够高，避免 overlay 位置漂移
  const workingHeight = Math.max(result.length, termHeight, minLinesNeeded);
  while (result.length < workingHeight) result.push("");

  const viewportStart = Math.max(0, workingHeight - termHeight);

  for (const { overlayLines, row, col, w } of rendered) {
    for (let i = 0; i < overlayLines.length; i++) {
      const idx = viewportStart + row + i;
      if (idx >= 0 && idx < result.length) {
        result[idx] = this.compositeLineAt(result[idx], overlayLines[i], col, w, termWidth);
      }
    }
  }
  return result;
}
```

两条技巧：

### 5.1 先把虚拟缓冲 padding 到 `termHeight`

这样"第 row 行"指的是**当前可见视口里的第 row 行**，overlay 位置稳定。不 padding 的话，内容只有 3 行时，overlay 想要定位在"屏幕中间"就会失败（下面根本没行可以写）。

### 5.2 `compositeLineAt` 单次扫描拼接 before / overlay / after

```ts
const base = extractSegments(baseLine, startCol, afterStart, totalWidth - afterStart, true);
const overlay = sliceWithWidth(overlayLine, 0, overlayWidth, true);

const result = base.before + " ".repeat(beforePad) + r + overlay.text + " ".repeat(overlayPad) + r + base.after + " ".repeat(afterPad);
```

ANSI 里一行字符串的视觉宽度 ≠ `line.length`：每个 `\x1b[31m` 都是字符但不占宽。pi-tui 在 `utils.ts` 里实现了 ANSI-aware 的 `visibleWidth`/`sliceByColumn`/`extractSegments`——是这套 overlay 合成能靠谱工作的基础。

### 5.3 一条防御性托底

合成后再做一次宽度检查：

```ts
const resultWidth = visibleWidth(result);
if (resultWidth <= totalWidth) return result;
return sliceByColumn(result, 0, totalWidth, true);
```

ANSI 序列太复杂，即使每步都小心算，也可能因为"宽字符在边界"或复杂 OSC 序列让最终视觉宽度偏离预期。最后一步强制截断到 `termWidth`，是**绝不能让超宽行送到终端去的防线**（超宽会换行 → 打乱所有光标计算 → 下次 diff 全错）。

## 6. `CURSOR_MARKER` 提取：差分 / 合成后才找光标位置

想象一个场景：你的 editor 组件在它的 render 里带了"光标占位符"（用于 IME，第 10 篇会专讲）。如果这个占位符在一帧的最后才被算出来，**它的 row 是合成后的 row**，不是组件自己知道的。

pi-tui 的顺序是：

```ts
let newLines = this.render(width);                              // 1. 走 Container.render 拿所有行
if (this.overlayStack.length > 0) newLines = this.compositeOverlays(...);  // 2. 叠 overlay
const cursorPos = this.extractCursorPosition(newLines, height); // 3. 扫光标标记
newLines = this.applyLineResets(newLines);                      // 4. 给每行追加 SGR reset
```

`CURSOR_MARKER` 是一个零宽的 `\x1b_pi:c\x07` APC 序列。`extractCursorPosition` 找到它在哪一行哪一列，然后把它从字符串里移除，光标位置信息单独返回。

这种"标记先放 render output 里 + 事后统一提取"的设计简直是神来之笔：**组件不需要知道"屏幕上的自己"在哪，只要把 marker 放在正确的字符位置即可**。overlay 盖上来也没事，marker 要么还在要么被遮住（找不到就隐藏硬件光标）。

## 7. 宽度超标立刻 crash 而不是"忍一下"

```ts
if (!isImage && visibleWidth(line) > width) {
  // ... 写 crash log
  this.stop();
  throw new Error(`Rendered line ${i} exceeds terminal width (${visibleWidth(line)} > ${width}).`);
}
```

自定义组件如果漏了 truncate，这一步会**立刻 crash + 写详细调试文件到 `~/.pi/agent/pi-crash.log`**。为什么不宽容？

- 宽行会让终端自动换行，破坏"每 string[] 一项对应一行"的不变式。
- 一旦破坏，后续所有差分计算都是错的，更新会漂移，非常难调试。
- 作者必须用 `truncateToWidth` / `wrapTextWithAnsi` 自觉遵守约定。

Fail fast + 良好 crash 日志 是优秀框架不可妥协的原则。

## 8. 可以直接抄走的套路

1. **请求驱动 + nextTick 合并 + 16ms 节流**：避免 render 风暴。
2. **把 UI 先渲染到 `string[]` 虚拟缓冲**，diff 后只写改动行。
3. **CSI 2026 包住一整帧**：原子输出消除撕裂，不支持也无害。
4. **首次渲染不清屏**：给用户保留 shell scrollback。
5. **宽度变化 / viewport 上移等极端场景走 fullRender**：diff 在这些情况不可靠。
6. **只画 `firstChanged..lastChanged`**：避免无谓的后续重绘。
7. **光标位置用 APC marker + 事后提取**：组件不需要知道屏幕坐标。
8. **超宽行立刻 crash + 写日志**：不妥协的不变式。

这些套路适用于任何"基于字符的 UI"项目：终端、文本编辑器、eBook reader、贪吃蛇 🐍、甚至基于画布的文本层。diff + 同步输出 + 虚拟缓冲 这三件套是所有"看起来平滑"的文本界面背后的通用套路。
