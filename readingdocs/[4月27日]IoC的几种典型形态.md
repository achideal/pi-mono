# IoC 的几种典型形态

> 配套阅读：[[4月27日]Agent与agent-loop分层设计-IoC与DI详解.md](./[4月27日]Agent与agent-loop分层设计-IoC与DI详解.md)
>
> 本文目标：把 IoC 拆成两种可操作的形态（**形态 A：依赖/策略反转** 与 **形态 B：主流程反转**），辨析二者差异、接口所有权归属，并定位本项目 `agent-loop` 在这张光谱上的具体位置。

---

## 1. 传统编码：没有明显 IoC

传统编码中，代码自己决定大部分控制点：

```text
传统编码（无明显 IoC）

┌─────────────────────────────────────┐
│ 我的代码决定一切                     │
│                                     │
│  • 用什么依赖：自己 new / 自己创建   │
│  • 使用哪个实现：自己写死            │
│  • 某个步骤怎么做：自己写死逻辑      │
│  • 何时启动：自己 main               │
│  • 何时调用谁：自己安排调用顺序      │
└─────────────────────────────────────┘
```

示例：

```python
class Agent:
    def __init__(self):
        self.llm = OpenAIClient()      # 自己决定用哪个依赖
        self.tools = [SearchTool()]    # 自己创建工具

    def run(self, user_input):
        response = self.llm.chat(user_input)  # 自己决定何时调用
        return response
```

这里的特点是：

> 依赖创建、实现选择、流程编排、调用时机，都由当前代码自己掌握。

## 2. 形态 A：把"用什么 / 怎么做"反转出去

这种情况下，**主流程仍然在我手里**，但某些可变决策交给外部。

```text
形态 A：依赖 / 策略 / 实现选择被反转，主流程仍在我

┌───────────────────────────────────────────┐
│ 我的代码                                  │
│                                           │
│  • 用什么依赖   ──→ 外部注入      ← 反转  │
│  • 使用哪个实现 ──→ 外部选择      ← 反转  │
│  • 某步骤怎么做 ──→ 外部策略      ← 反转  │
│  • 何时启动     ←── 我自己控制            │
│  • 何时调用谁   ←── 我自己控制            │
└───────────────────────────────────────────┘
```

> **接口由"我"定义，外部来实现。**
>
> "我规定 `LLMClient` 应该长什么样，你来满足我。"

```text
    我 ──定义接口──▶ ILLMClient
                        ▲
                        │ 实现
                        │
                  外部依赖（OpenAI / Claude / Local...）
```

典型例子：

- 依赖注入（Dependency Injection）
- 策略模式（Strategy）
- 插件机制
- 配置化
- 回调函数注入
- 本项目的 `agent-loop` 设计

### 对照 `agent-loop` 的自我定位

套用到本项目：

| 控制点 | `agent-loop` 的态度 | 反转 |
|:---|:---|:---:|
| 用什么依赖 | 不关心（`tools`、`streamFn`、`hooks` 全从外部来） | ✅ |
| 怎么执行 | 某些步骤允许策略替换（`toolExecution` 模式、`beforeToolCall`/`afterToolCall`） | ✅ |
| 何时启动 | 由外部（`Agent` 或其他调用方）主动触发 | ❌ |
| 何时调用谁 | **自己掌握主流程时序** | ❌ |

结论：**`agent-loop` 是标准的形态 A（DI / 依赖倒置）**。

它**不是** Spring Boot / React 那种"框架反过来叫你"的形态 B，它自己不是框架，外壳（`Agent`）才是调用它的一方。

> 主流程还在自己手里，但部分控制点被外部化了。

## 3. 形态 B：把"何时启动 / 何时调用"反转出去

这种情况下，重点不是依赖怎么创建，而是：

> **主流程不再由我掌握，而是交给框架。**

```text
形态 B：主流程被反转给框架

┌─────────────────────────────────────────────────┐
│ 我的代码                                        │
│                                                 │
│  • 用什么依赖  ←── 可能自己管理，也可能注入     │
│  • 具体怎么做  ←── 我提供业务实现               │
│  • 何时启动    ──→ 框架控制              ← 反转 │
│  • 何时调用我  ──→ 框架控制              ← 反转 │
│  • 调用顺序    ──→ 框架控制              ← 反转 │
└─────────────────────────────────────────────────┘
```

典型例子：

- Web 框架路由处理函数
- GUI 事件回调
- 测试框架（如 pytest）
- Servlet
- React 组件
- Vue 组件
- 生命周期 hook

示例：Flask

```python
from flask import Flask

app = Flask(__name__)

@app.route("/hello")
def hello():
    return "hello"
```

你没有自己调用：

```python
hello()
```

而是 Flask 在合适的时候调用你：

```text
HTTP 请求进来
    ↓
Flask 监听端口
    ↓
Flask 解析请求
    ↓
Flask 匹配路由
    ↓
Flask 调用 hello()
    ↓
Flask 生成响应
```

这里的反转是：

```text
传统模式：我调用库
IoC 框架：框架调用我
```

> "框架规定 `Component` 应该长什么样，我来满足它。"

```text
  框架 ──定义接口──▶ Component / Handler / Hook
                        ▲
                        │ 实现
                        │
                      我的代码
```

也就是所谓的 Hollywood Principle：

> Don't call us, we'll call you.
> 不要调用我们，我们会调用你。

## 4. 形态 A + B：现代框架中最常见的组合

很多现代框架同时具备两类反转：

1. **依赖 / 实现选择反转**
2. **主流程 / 调用时机反转**

```text
形态 A + B：依赖选择和主流程都被反转

┌──────────────────────────────────────────────┐
│ 我的代码                                     │
│                                              │
│  • 用什么依赖   ──→ 外部注入      ← 反转 A   │
│  • 使用哪个实现 ──→ 外部配置      ← 反转 A   │
│  • 具体怎么做   ←── 我提供实现               │
│  • 何时启动     ──→ 框架控制      ← 反转 B   │
│  • 何时调用我   ──→ 框架控制      ← 反转 B   │
│  • 调用顺序     ──→ 框架控制      ← 反转 B   │
└──────────────────────────────────────────────┘
```

典型例子：

- Spring Boot
- NestJS
- Angular
- ASP.NET Core
- 一些大型后端框架
- 结合 DI 容器使用的 Flask/FastAPI 项目

示例：Spring Boot

```java
@RestController
public class UserController {

    private final UserService userService;

    // 依赖由 Spring 注入：形态 A
    public UserController(UserService userService) {
        this.userService = userService;
    }

    // 调用时机由 Spring 控制：形态 B
    @GetMapping("/users/{id}")
    public User getUser(@PathVariable Long id) {
        return userService.findById(id);
    }
}
```

这里同时发生了两种 IoC：

```text
形态 A：
UserController 不自己 new UserService，
而是由 Spring 注入。

形态 B：
UserController 不自己监听 HTTP 请求，
而是由 Spring 在请求到来时调用 getUser()。
```

## 5. React 属于哪一种？

React 主要属于 **形态 B：主流程反转**。

你写的是：

```jsx
function App() {
  const [count, setCount] = useState(0)

  return (
    <button onClick={() => setCount(count + 1)}>
      {count}
    </button>
  )
}
```

你并不控制：

```text
• App 什么时候被调用
• 组件什么时候重新渲染
• state 更新什么时候被处理
• DOM 什么时候被修改
• effect 什么时候执行
• cleanup 什么时候执行
• 组件什么时候 mount / unmount
```

这些由 React 控制：

```text
React 控制：

事件分发
    ↓
状态更新排队
    ↓
调度优先级
    ↓
重新调用组件函数
    ↓
diff
    ↓
commit 到 DOM
    ↓
执行 effect / cleanup
```

所以 React 中的 IoC 是：

```text
我提供组件、hook、事件处理函数
React 决定什么时候调用它们
```

也就是：

```text
我的代码：声明 UI 应该是什么样
React：决定何时渲染、如何渲染、如何更新 DOM
```

不过，React 本身不以 DI 为核心。
当然，你可以通过 Context、Provider、props、hooks 等方式实现某种依赖外部化，但那不是 React 的核心 IoC 形态。

## 6. 两种形态的核心区别

| 维度 | 形态 A：依赖 / 策略反转 | 形态 B：主流程反转 |
|---|---|---|
| 主流程在谁手里 | 在我的代码手里 | 在框架手里 |
| 我扮演什么角色 | 调用方、流程编排者 | 被框架调用的组件 / handler / hook |
| 反转的是什么 | 依赖选择、实现选择、步骤细节 | 启动时机、调用时机、生命周期、流程编排 |
| 我声明什么 | 我需要什么能力 | 我提供什么能力 |
| 外部做什么 | 给我依赖、策略、实现 | 在合适时机调用我 |
| 典型机制 | DI、策略模式、插件、配置 | Framework、事件循环、生命周期、hook |
| 例子 | `agent-loop`、策略注入、插件系统 | React、Flask、GUI、pytest |

## 7. 接口所有权的区别

还可以从"接口由谁定义"这个角度理解。

### 形态 A：接口通常由"我"定义，外部来满足

```text
形态 A：我定义需求，外部提供实现

我的代码：
    "我需要一个 LLMClient，
     它只要有 chat(messages) 方法即可。"

外部：
    "好，我可以给你 OpenAIClient，
     也可以给你 ClaudeClient，
     也可以给你 MockLLMClient。"
```

示例：

```python
class LLMClient(Protocol):
    def chat(self, messages: list[dict]) -> str:
        ...


def run_agent_loop(llm: LLMClient, messages):
    return llm.chat(messages)
```

这里更接近：

```text
Dependency Inversion Principle（DIP）
Ports and Adapters
依赖抽象，而不是依赖具体实现
```

注意：

> DI 是把依赖传进来的机制；
> DIP 是高层模块依赖抽象、不依赖低层细节的设计原则。
> 二者经常一起出现，但不是同一个东西。

---

### 形态 B：接口通常由"框架"定义，我来实现

```text
形态 B：框架定义扩展点，我提供实现

框架：
    "组件必须是一个函数。"
    "路由 handler 必须接收 request，返回 response。"
    "hook 必须按我的规则调用。"
    "测试函数必须以 test_ 开头。"

我的代码：
    "好，我按你的规则写。"
```

示例：React

```jsx
function App() {
  return <div>Hello</div>
}
```

示例：pytest

```python
def test_add():
    assert 1 + 1 == 2
```

示例：Flask

```python
@app.route("/hello")
def hello():
    return "hello"
```

这里的核心是：

```text
框架定义协议
我实现协议
框架在合适的时机调用我
```

## 8. 总结

```text
IoC 的本质不是"谁调用谁"这么简单，
而是：原本由自己写死的控制点，被外部化了。

这些控制点可以是：

1. 用什么依赖
2. 使用哪个实现
3. 某个步骤怎么执行
4. 事件发生后怎么处理
5. 何时启动
6. 何时调用谁
7. 主流程如何编排
8. 生命周期如何管理
```

可以分成两种典型形态：

```text
形态 A：我仍然掌握主流程，但把依赖、策略、实现选择交给外部
       例：DI、策略模式、插件机制、agent-loop

形态 B：我把自己写成组件、handler、hook、回调，
       由框架决定何时调用我
       例：React、Flask、GUI、pytest

形态 A + B：现代框架常见组合
       例：Spring Boot、NestJS、Angular、ASP.NET Core
```

一句话：

> **IoC 不是单一概念，而是一个关于"反转了哪个控制点"的连续光谱。形态 A（DI / 依赖倒置）反转的是"用什么依赖、怎么执行"，接口由我定义、外部实现；形态 B（Hollywood Principle）反转的是"何时启动、何时调谁"，接口由框架定义、我来实现。现代主流框架通常是 A+B 叠加，而 `agent-loop` 属于纯粹的形态 A。** 使用现代框架开发时，通常会**同时使用这两种反转**。
